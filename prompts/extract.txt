You are converting an already-gathered set of verbatim quotes from a research paper into a structured JSON record for downstream meta-analysis. A previous stage pulled the quotes; your job is to organize them into a clean schema. You should NOT introduce new facts that are not in the quotes — if a piece of information is not in the quotes, the corresponding field must be `null` (or an empty list).

CRITICAL RULES
1. Use ONLY information present in the provided quotes. Do not invent, infer beyond what is plainly stated, or supplement from memory.
2. If a field is not supported by the quotes, set it to `null` (scalars) or `[]` (lists). Do not guess.
3. For numeric values, copy them exactly as they appear in the quote — do not round, normalize, or convert units.
4. **p-values**: report the EXACT reported value as a string. Examples: `"p = 0.021"`, `"p = 0.003"`, `"p < .001"`, `"p < .01"`, `"p > .05"`. If only an inequality (`p < .01`, `p < .05`) is given, that is what you record — do NOT replace it with an exact-looking value, and do NOT round. If no p-value is reported for a comparison, the `p_value` field is `null`. A rounded threshold like `p < .01` is acceptable as-is; an invented exact value is NOT. **NEVER write prose labels like `"significant"`, `"not significant"`, `"NS"`, `"marginal"`, or asterisks-without-a-number — those are not p-values and must be recorded as `null` (with a note in `notes` if relevant).** The only acceptable values are: a literal `p = X` numeric, a literal `p < X` / `p > X` inequality, or `null`.
5. Extract ALL outcomes and ALL reported statistics that appear in the quotes — filtering happens downstream.
6. If the paper has >2 arms (e.g., no-AI, GPT-3.5, GPT-4), include ALL arms.
7. Each `reported_statistics` entry MUST include a `verbatim_quote` field copied directly from the input quotes. This is the audit trail.
8. Output a single JSON object. No prose, no markdown fences.
9. **JSON escaping** (CRITICAL): every double-quote character inside a JSON string value MUST be escaped with a backslash. For example, the sentence containing a "linear" scheme must be written as "a \"linear\" scheme". Do not substitute curly quotes or apostrophes. Escape backslashes as \\ and newlines as \n.

SAMPLE SIZES — READ CAREFULLY, LEAVING THESE NULL WHEN AVAILABLE IS THE #1 EXTRACTION ERROR
- Arm-level sample size (`arms[].n`) is the count of participants/units assigned to that arm. Each statistic in `reported_statistics` implicitly inherits this n from the arms it compares — do NOT repeat per-statistic n inside `means_and_sds`. Downstream stages will look up n from `arms[].n`.
- **`arms[].n` MUST be filled in if the quotes contain a sample size attributable to that arm anywhere.** Before deciding `n = null`, you MUST check every topic in the input quotes: `study_design`, `task`, `population`, `arms`, AND `statistical_results`. Sample sizes are commonly mentioned in `population` ("444 participants randomized 50/50") or `study_design` ("N = 222 treatment, N = 222 control") rather than the `arms` topic specifically.
- If only a total sample size is given (e.g., "n = 444 participants randomized 50/50"), split it across the arms by the stated allocation. Specifically: "randomized 50/50" → 222/222. "randomly assigned half to treatment" → split evenly. "stratified random sample of 200 per arm" → 200 per arm.
- For panel/observational studies, the n is the number of UNITS in each arm (developers, firms, projects), not the number of observations. If the paper says "1187 R packages, 1373 Haskell packages" those ARE the arm ns even if the regression uses package-quarter observations.
- If a paper reports a different n for a specific analysis (e.g., listwise deletion left fewer participants in one model), capture that in `notes` rather than restructuring the arms.
- Only set `n = null` when you have read all the quotes and confirmed no sample size is stated anywhere.

EFFECT-SIZE FIELDS
- `effect_in_outcome_units`: the comparison expressed in the original measurement scale. E.g., "26.08% faster" (time-based outcome), "saved 5 minutes", "12 more tasks per hour", "Δ = 4.2 rubric points", "OR = 1.34".
- `standardized_effect_reported`: a paper-reported standardized effect (unit-free). E.g., "Cohen's d = 0.41", "Hedges' g = 0.55", "0.5 SD increase", "η² = 0.08". These are values that should NOT be recomputed downstream; they're papers' own derivations.
- `regression_coefficient`: a regression-style coefficient with its standard error. This is the dominant effect-size format for natural experiments (DiD, RD, IV) and many observational studies. Look for tables showing "Treated × Post" coefficients, "β" coefficients with SE in parentheses, fixed-effects regression results, etc. Always include the SE (typically in parentheses right after the point estimate), and note the outcome transformation if mentioned ("log", "asinh", "level"). EXAMPLES of source text → field value:
    "Treated×Post  0.0062 (0.0076)  for Output" → "Treated×Post = 0.0062 (SE 0.0076), outcome: Output"
    "β = -0.045**, robust SE in parentheses (0.022), n = 50,000" → "β = -0.045 (SE 0.022), **"
    "DiD coefficient on log(commits): 0.123 (0.041)" → "DiD coef = 0.123 (SE 0.041), outcome: log(commits)"
- All three fields are independent — fill in any combination the paper reports.
